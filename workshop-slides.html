What is
Resilience?

The Problem

User

One slow or failing service can take down your entire system.

Cascading Failure

❌ Without Resilience

  • Service B slows down
  • Service A threads wait...
  • Thread pool exhausted
  • All requests blocked
  • Service A is DOWN

✓ With Resilience

  • Service B slows down
  • Circuit breaker trips
  • Service A fails fast
  • Fallback: show cached data
  • Service A stays UP

Resilience Patterns We'll Learn

1. Retry

  • Try again when things go wrong
  • With backoff + jitter
  • Handles transient failures

2. Circuit Breaker

  • Stop calling what's broken
  • Fail fast, use fallback
  • Handles sustained failures

3. Distributed Retry

  • Retry even when you crash
  • Persistent queue + scheduled job + distributed lock

Phase 1

The Chaos

Observe the Disaster

No code changes. Just observe.

  1. Start both services (mvn spring-boot:run)
  2. Call GET /products — verify it works
  3. Toggle Service B to slow mode
  4. Call GET /products from multiple tabs
  5. Toggle Service B to error mode
  6. Call GET /products again
  7. Bonus: docker stop pricing-service

What Did You Observe?

⏱️ Slow Mode

  • Service A hung for 10+ seconds
  • All threads blocked
  • Entire service became unresponsive

❌ Error Mode

  • 500 errors propagated to users
  • No graceful degradation
  • No fallback
Key insight: A slow service is worse than a dead service.
Dead = fails fast. Slow = holds your threads hostage.

Challenge 1: Add Basic Retry

  1. Toggle Service B: curl -X POST "localhost:8081/admin/random?rate=40"
  2. Hit GET /products — you see failures
  3. Add @Retry(name = "pricingService") to PricingClient.getPrice()
  4. Configure in application.yml
@Retry(name = "pricingService")
public Map<String, Object> getPrice(String productId) { ... }

resilience4j:
  retry:
    instances:
      pricingService:
        maxAttempts: 3
        waitDuration: 500ms

Exponential Backoff

❌ Without Backoff

  • All retries at 500ms
  • Creates load spikes
  • Can overload the failing service

✓ With Backoff

  • 500ms → 1s → 2s
  • Gives service breathing room
  • Better chance of recovery

Jitter: Prevent Thundering Herd

❌ Without Jitter

  • 10 clients all retry at same time
  • Spike at t=0.5s, t=1s, t=2s
  • All hit Service B simultaneously
  • Even worse than before!

✓ With Jitter

  • 10 clients retry at random intervals
  • Requests spread out
  • No thundering herd
  • Smooth distribution

Challenge 2: Fix the Retry Storm

Your retries are making things worse!

  1. Service B in slow mode + 10 requests = 30 calls to Service B!
  2. Add exponential backoff + jitter to the config
resilience4j:
  retry:
    instances:
      pricingService:
        maxAttempts: 3
        waitDuration: 500ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
        enableRandomizedWait: true
        randomizedWaitFactor: 0.5
Bonus: Only retry on 5xx errors — NOT 400 Bad Request!

The Circuit Breaker

The Circuit Breaker Pattern

CLOSED

Normal operation

→
failures > 50%

OPEN

Failing fast

→
wait 10s

HALF_OPEN

Testing

↻
succeed? → CLOSED
fail? → OPEN

CLOSED State (Normal)

Request comes in
↓
Call Service B
↓
Count success/failure
↓
After 10 calls, if failures > 50% → OPEN

OPEN State (Failing Fast)

Request comes in
↓
Circuit OPEN → Reject immediately
↓
Call fallback (no Service B call!)
↓
After 10s → HALF_OPEN

HALF_OPEN State (Testing)

3 test calls allowed through
↓
If ≥2 succeed (67%+) → CLOSED ✓
↓
If ≥2 fail → OPEN again ❌

Challenge 3: Add Circuit Breaker

  1. Toggle Service B: curl -X POST localhost:8081/admin/fail
  2. Send requests repeatedly — watch the circuit open
  3. Check state: GET /actuator/circuitbreakers
  4. Toggle back to healthy — watch it recover
@CircuitBreaker(name = "pricingService")
public Map<String, Object> getPrice(String productId) { ... }

resilience4j:
  circuitbreaker:
    instances:
      pricingService:
        slidingWindowSize: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 10s

The Fallback

When the circuit opens, give users something useful, not an error.

@CircuitBreaker(name = "pricingService",
                 fallbackMethod = "getPriceFallback")
public Map<String, Object> getPrice(String productId) { ... }

private Map<String, Object> getPriceFallback(
        String productId, Throwable t) {
    return Map.of(
        "price", cachedPrice,
        "priceStale", true,
        "error", "Price temporarily unavailable"
    );
}
The fallback method signature must match: same params + Throwable as last param.

The #1 Workshop Pitfall

⚠️ @CircuitBreaker doesn't work when you call the annotated method from within the same class!

❌ WRONG

@CircuitBreaker(name = "ps")
public void myMethod() {
    anotherMethod();  // internal call
}

public void anotherMethod() {
    // AOP proxy NOT used!
}

✅ RIGHT

@Service
public class ProductService {
    @Autowired
    private PricingClient client;

    public void myMethod() {
        client.getPrice(id);  // through proxy!
    }
}

Challenge 4: Combine Retry + Circuit Breaker

The order matters!

❌ Wrong Order

  • @Retry outer, @CircuitBreaker inner
  • Circuit opens → retries still happen
  • Wasted effort!

✅ Right Order

  • @CircuitBreaker outer, @Retry inner
  • Circuit open → skip retry
  • Go straight to fallback
resilience4j:
  circuitbreaker:
    circuitBreakerAspectOrder: 1   # outer
  retry:
    retryAspectOrder: 2             # inner

Wrap-up

Distributed Retry

The Problem with In-Memory Retry

Everything dies with the process

  • Service A crashes → retries lost
  • 5 instances → who retries what?
  • Failure lasts hours → give up too soon
We need persistent retry.

❌ Without ShedLock

  • 5 instances run scheduler → 5x retries!

✓ With ShedLock

  • 1 instance runs → exactly 1 retry
